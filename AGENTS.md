# AGENTS.md

「追いつかれた車両の義務」の有無が渋滞のできかたに与える影響を、L区間（義務あり）と R区間（義務なし）の 2 本を並べて比較する Three.js 製の渋滞シミュレーション。

---

## A. 最重要: 実験的妥当性の不変条件

**このプロジェクトの存在意義に直結する。何よりも先に守ること。**

- 本シミュレーションは **L区間（義務あり）** と **R区間（義務なし）** を並置し、「**追いつかれた車両の義務**」というルール 1 つの有無だけが渋滞に与える影響を比較する対照実験である。
- **両区間の挙動差は、この「追いつかれた車両の義務」ただ 1 つでなければならない。** それ以外に区間で異なる挙動を導入すると交絡（confounding）が生じ、比較そのものが無意味になる。
- **実装上の禁止事項:** 新しい挙動の分岐条件に、区間で値が変わる変数（`section` / `yields` / `camper` / `keepLeft` / `returnTime` など）を使ってはならない。唯一の例外は既存の「追いつかれた車両の義務」ロジック（`src/core/vehicle.ts` の `if (this.yields && flowing && this.lane < 2)` の箇所とキープレフト）である。新しい挙動は **L/R 完全同一（区間非依存）** で実装する。物理（加速・車間・合流など）は当然両区間共通。
- **回帰ガード:** `traffic-simulation.test.ts` のシードテストが L/R の渋滞スコア差「**約 10 ± 2**」（`DIFF_TARGET = 10`、10 シード平均）を強制する。これを壊さないこと。そして **その数値を回復するために区間依存の条件を足して辻褄を合わせる行為は禁止**。それはこの不変条件そのものの違反であり、本末転倒である。数値が区間非依存で回復できないなら、**実装を止めて人間に報告する**。
- **経緯（なぜこの規約があるか）:** かつて合流協調を `yields` でゲートしたところ、「義務」が合流にも二重に効いて交絡になり、レビューで是正された実例がある。区間差を生む変数に安易に相乗りしないこと。

## B. 車線・座標の規約

- **lane index の意味:**
  - `0` = 追い越し車線（進行方向の**右端**, `+X` 側）
  - `1〜2` = 走行車線
  - `3` = 加速車線（合流ランプ）
  - 実体は `src/core/constants.ts` の `LANE_X`（`LANE_X_L = [-3, -7, -11, -15]`）。index 0 が最も `+X`（右）、index 3 が最も外（左外側）。
- **R区間は L区間の鏡像ではなく平行移動コピー**（`SECTION_OFFSET_X`, 実値 `SECTION_OFFSET_R_X = 17.2`）。これにより追い越し車線は両区間とも右、**両区間の合流条件まで完全に同一**になる。鏡像にすると R だけ追い越し車線が左になり交絡するので不可。
- **進行方向は `-Z`**（前方 = `z` が小さい側）。合流ランプは `RAMP_Z_TOP`(380) から `RAMP_Z_END`(250) へ、z が減る向きに走る。

## C. 開発・検証コマンド

- ランタイム: **Node.js 26**（`.node-version` で固定 / `package.json#devEngines` が `>=26.0.0`）。
- パッケージマネージャ: **pnpm**（`package.json#packageManager` = `pnpm@11.13.0`）。
- ツールチェーンは **Vite+（`vp`）** に統一（ビルド=Vite / テスト=Vitest / lint=Oxlint / フォーマット=Oxfmt / 型チェック）。設定は `vite.config.ts` に集約。

| コマンド                    | 内容                                                                                   |
| --------------------------- | -------------------------------------------------------------------------------------- |
| `pnpm install`              | 依存導入 + git hooks 導入（`prepare` で `vp config` が走る）                           |
| `pnpm check`                | **フォーマット + lint + 型チェックを一括実行**（`vp check`）。コミット前の基本チェック |
| `pnpm test`                 | シミュレーションコアのテスト（Vitest, `vp test run`）                                  |
| `pnpm build`                | `dist/` へバンドル出力                                                                 |
| `pnpm lint` / `pnpm fmt`    | lint のみ / フォーマット適用                                                           |
| `pnpm dev` / `pnpm preview` | 開発サーバー / ビルド結果確認サーバー                                                  |

- 依存（three / lucide）は pnpm で管理し Vite+ でバンドルする。**CDN は使わない。**
- コミット時は `.vite-hooks` の pre-commit フックが `vp staged`（= ステージ済みへ `vp fmt`）を適用する。

## D. テスト規約

- テストは単一ファイル **`traffic-simulation.test.ts`**（`src/core/` の DOM/THREE 非依存なシミュレーションロジックを検証）。
- 構成は**内容ごとの `describe` セクション**（例: `追いつかれた車両の義務`、`追い越し`、`流入・流出と滞留 (Issue #12)` など。Issue 由来のものは末尾に `(Issue #N)` を付ける運用）。
- **新規テストはファイル末尾に、既存セクションと重複しない新しい `describe` ブロックとして追加する。** 並行作業でファイル末尾が衝突しやすいので、追加前に既存の `describe` 一覧を確認すること。
- スコア差の回帰ガード（A 参照）を含むため、テストの期待値を都合よく緩めないこと。
